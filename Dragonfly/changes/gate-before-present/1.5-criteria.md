# 1.5 — Acceptance criteria

Prose change to two methodology docs → no measurable prior metric → **stage 8 conformance-only**
(no stage-0 baseline). Checks are text/structure inspection plus one **execution (replay)** that is
load-bearing: does the new rule, applied to the documented slip, actually forbid the thing that
went wrong?

| # | Criterion | Gating? | How verified (stage 8) |
|---|---|---|---|
| **C1** | The **gate-before-present rule** is stated in all load-bearing locations so it fires: (a) a Core Principle, (b) a stage-3 (Hypotheses) stage-detail rule, (c) an entry in Stop-for-human / Human-in-the-loop, and (d) a SKILL.md operating-procedure clause. | **gating** | Locate each in the edited docs. A rule present in only one place is easy to skip; the slip happened at presentation time, which the Human-in-the-loop + stage-3 locations both govern. |
| **C2** | The rule explicitly forbids presenting a hypothesis to the human as **"leading/likely/probable/most-likely cause"** (or a conclusion to act on) **until it has passed its cold red-team**, and names the pre-gate state **"candidate, ungated."** It also explicitly **permits** showing a ranked candidate list (rank ≠ endorsement). | **gating** | Text names both the prohibition (with the label list) and the explicit carve-out that ranking/among-candidates presentation is allowed — so the rule can't be read as "never discuss hypotheses." |
| **C3** | **Per-hypothesis gate-coverage marker** is added to the `hypotheses.md` schema: an explicit gate status (e.g. `ungated → test-passed → cold-red-teamed`) distinct from `open/confirmed/refuted`, and the stage-7 "confirmed" bar is tied to `cold-red-teamed`. | **gating** | The schema description in METHODOLOGY (stage 3 / "What a run produces") and SKILL show the marker and its values, and stage 7 references it. |
| **C4** | **Trust-before-gate ordering rule**: an artifact's output may not be consumed by a later stage until its triage (representativeness gate + cold review) is recorded as passed — stated as a precondition of trust, not a retroactive audit. | **gating** | Text presence in METHODOLOGY (stage 5 "Run & record" and/or the triage/representativeness section) and a SKILL clause. |
| **C5** | **REPLAY (execution, discrimination not recall).** Given *only the new rule text* + THREE neutral scenarios (a fact pattern **different** from the example written into Edit B, per R1), a cold subagent classifies each correctly: **(V) violation** — a hypothesis with a *passed, representative discriminating test* but **no stage-7 causal red-team**, presented to the user as **"the root cause, let's fix it"**; **(A1) compliant** — same hypothesis presented as **"the leading candidate so far, causal chain not yet independently checked"**; **(A2) compliant** — a ranked candidate list shown with none called the cause. | **gating** | Cold subagent (no shared context) given the rule text + the three scenario descriptions must return V=violation, A1=compliant, A2=compliant. Pass = all three correct. This tests the rule's *discriminating force* (the V/A1 boundary is exactly the tier line — root-cause claim needs `cold-red-teamed`, leading-candidate needs only `test-passed`), not recall of the motivating example, and confirms it neither under-fires (V) nor over-fires (A1/A2). |
| **C6** | The change does **not** contradict the existing `open/confirmed/refuted` status, the stage-7 three-part "found" bar, or the "form/rank hypotheses freely in working notes" latitude. | advisory | Stage-6 cold review reports no logical inconsistency; the gate marker refines rather than replaces the status. *Advisory:* coherence, minor-in-place fixable. |
| **C7** | Both copies (live `~/.claude/skills/dragonfly/` + source `~/Desktop/claude-code-skills/Dragonfly/`) updated identically. | **gating** | `diff` the two copies of each changed file → empty. |

**Note on C5 design (learned from session 1):** the replay must NOT pre-digest the answer. The cold
judge gets the rule + neutral scenario facts and must itself decide violation/compliance — including
the **compliant control** (ranked list, no endorsement), so a rule that simply bans all hypothesis
talk would FAIL C5 by flagging the control too. This is the two-sided representativeness check
(catches the bug AND spares the legitimate case).
