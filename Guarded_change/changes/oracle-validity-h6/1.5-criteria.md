# 1.5 — Criteria (the accept bar stage 8 checks)

All criteria are **gating** unless labelled advisory. Verification means = how stage 8 proves it
(by execution, not by re-reasoning about the text). **Dogfood note:** these criteria's own oracles
follow the very rule being added — every oracle below is a **positive per-site assertion**, and any
grep over prose is **flattened first** (`tr '\n' ' '`, and `**`-stripped where markup could
interfere) so a line-wrap or emphasis cannot defeat it; every non-degenerate oracle carries a
**fire-on-a-known-violating-input self-test** so the oracle itself is shown able to fail. (Round-1
red-team caught that the first-draft C3 grep was itself wrap-blind — see 3-redteam-plan.md; C3 is
redesigned below to the standard it preaches.)

## Content criteria (the rule text)

- **C1 (gating) — H6 generalized to all conformance oracles.** After the change, H6 in
  `stages/stage-8.md` states that the "must be shown able to fail / demonstrated to fire on a
  known-violating input before its pass is trusted" duty applies to **any conformance/oracle check**
  whose result stage 8 trusts — explicitly including a cheap grep/diff that guards no fix — not only
  a check guarding a behavioral fix. The phrase "an oracle that cannot be shown to fail is an un-run
  check" (or equivalent) is present. *Verify (positive assertion):* the built `stage-8.md` H6
  paragraph contains a clause generalizing beyond fix-guarding checks AND the self-test/known-
  violating-input language. Judge = run operator (human-judged rubric: both sub-parts locatable).

- **C2 (gating) — Positive-assertion-over-absence preference present in H6.** H6 states a preference
  for a **positive per-site assertion** over a bare **absence check**, gives the reason (a fragile
  matcher makes "absent" silently/wrongly true), and requires that where an absence sweep is used it
  be **paired with the positive assertion AND text normalized first** (strip markup / flatten wraps).
  *Verify (positive assertion):* the built H6 paragraph(s) contain all three elements — (a) prefer
  positive per-site, (b) the silent-false-absence rationale, (c) pair-with-positive + normalize.
  Judge = run operator (human-judged rubric: (a)–(c) locatable). Pass = all three present.

- **C3 (gating, REDESIGNED per round-1 BLOCKER) — H6's original fail-against-unguarded clause
  preserved intact — wrap-robust positive assertion.** The pre-existing H6 sentence about a
  fix-guarding check failing against the **unguarded version** still reads as before (the two
  additive spans are appended after H6's original sentences, so this clause is unmoved). A bare
  `grep -c 'fails against the unguarded version' stage-8.md` is a **broken oracle**: the clause is
  line-wrapped in source (`…the`\n`unguarded version…`) so it returns 0, and the delete-phrase
  self-test is degenerate (baseline already 0). C3 is therefore a **flatten-then-assert** positive
  check: `tr '\n' ' ' < stages/stage-8.md | grep -c 'fails against the unguarded version'` must
  equal **1**. *Self-test (mandatory, stage 8):* run the *same* flattened grep against a scratch
  copy of the built file with that clause deleted and confirm it returns **0** — the oracle fires
  on a known-violating input (replacing the degenerate baseline self-test).

- **C4 (gating) — stage-1.5 verification-method note added.** `stages/stage-1.5.md` gains a note,
  at the point where a criterion states its verification method, that a criterion's oracle must be
  shown able to fail (self-test that it fires on a known-violating input) before a pass counts, and
  that a positive per-site assertion is preferred over a bare absence check. It references H6 as the
  stage-8 enforcement. *Verify (positive assertion):* the built `stage-1.5.md` contains the note with
  (a) oracle-must-be-shown-able-to-fail, (b) prefer-positive-over-absence, (c) an H6 reference.
  Judge = run operator (human-judged rubric: (a)–(c) locatable). The note is added as rule
  **ST1.5f**, placed immediately after ST1.5a (the checkability rule it extends); its ID letter is
  out of alphabetical order (a, f, b, …) but IDs are labels, not a sequence — semantic placement
  wins (round-1 M1, accepted with this note).

## Behavior-preservation / no-collateral-edit criteria (position lens)

The skill docs are prompts — a **position-sensitive assembly**. The change is **additive**: append
two spans to the END of the H6 paragraph in stage-8.md (so H6's original sentences are byte-unmoved)
+ insert a new rule note ST1.5f in stage-1.5. Behavior to preserve: every other rule in both files
reads and applies exactly as before, AND H6's original sentence adjacency is intact (see C10).

- **C5 (gating) — No other H-rule / CP / SEV / CH rule text altered.** In `stage-8.md`, every rule
  block **other than H6** (H1–H5, H7, H8, the CP/FRZ/SEV/ART/CH/HIL blocks) is byte-unchanged, and
  H6's own original text is unchanged up to the appended spans; in `stage-1.5.md`, every existing
  rule (CP4/CP5, ST1.5a–e, CP6, CP7, CFG4/CFG5, HIL, the self-check block) is byte-unchanged except
  for the inserted ST1.5f note. *Verify by execution:* `git diff` hunks touch **only** the tail of
  the H6 paragraph in `stage-8.md` (an append) and **only** the inserted ST1.5f lines in
  `stage-1.5.md` — no hunk lands inside any other rule block or inside H6's original sentences.
  Positive form: enumerate every diff hunk and confirm each falls within the two allowed edit
  regions (H6-append; ST1.5f-insert).

- **C6 (gating) — The diff is exactly {extend H6} ∪ {add stage-1.5 note} and nothing else.** No
  file other than `stages/stage-8.md` and `stages/stage-1.5.md` is modified (changes/ folder
  excepted). *Verify:* `git status --short` shows only those two tracked files modified (plus new
  untracked `changes/oracle-validity-h6/`); `git diff --stat` confirms the two files.

- **C7 (gating) — position preservation earned by a cold reader.** A stage-6 cold reviewer, reading
  the edited `stage-8.md`, confirms it can still enumerate and apply H1–H5, H7, H8 and the CP/SEV/CH
  rules unchanged AND now also the generalized H6 — i.e. the extended H6 block did not displace or
  reword a load-bearing neighbor. (The "behavior" of a prompt-doc rule is that a reader can
  enumerate and apply it; the reviewer's enumeration is the execution.) *Verify:* stage-6 record.

## Cross-doc consistency criteria

- **C8 (gating, REWORDED per round-1 L1) — No stale cross-doc reference to H6's old scope.** H6 the
  oracle rule is stated in exactly one place (the `**An unreviewed check is not a check — H6.**`
  block in `stage-8.md`); no *other* skill file restates H6's scope, so generalizing it creates no
  cross-doc inconsistency. The only new mention is the ST1.5f note's **reference** to H6 (a pointer,
  like stage-1.5 already references CP6/CP7), which must not contradict the generalized rule.
  *Verify:* positive assertion — `grep -rn 'unreviewed check' --include='*.md'` over the source tree
  (excluding `changes/`) returns **exactly one** hit, the H6 rule-statement line in `stage-8.md`
  (this is a stable, wrap-safe anchor — the phrase is contiguous). The ST1.5f reference is a pointer
  and is checked by C4, not this grep. (No degenerate self-test needed: the pattern is exact and the
  single-hit count is the assertion. Note — deliberately NOT `grep 'H6.'`: an unescaped `.` is a
  regex wildcard that also matches `CH6,`/`CH6)`, so it would wrongly report the two unrelated CH6
  rule sites; the earlier draft's "two CH6 hits" description was that mistake, now removed.)

## Standing self-check criteria (mandatory when the artifact is this skill)

- **C9 (gating) — live copy == source copy.** After the build + sync, `diff -r` of the installed
  skill (`~/.claude/skills/guarded-change/`) against the source
  (`~/Desktop/claude-code-skills/Guarded_change/`, excluding `README.md`, `*.companion.md`,
  `changes/`) is empty. *Verify:* `diff -r` exit 0 (positive: the two edited files are byte-identical
  across source/installed).

- **C10 (gating, NEW per round-1 MAJOR — position lens, intra-rule anaphora) — H6's original
  sentence adjacency preserved.** The `A defective check found this way…` sentence must still sit
  **immediately after** its referent — H6's original targeted-cold-check sentence ending
  `until then its results are \`verified = no\`.` — so the anaphor "this way" keeps binding to the
  targeted cold check (the two additive spans are appended AFTER this pair, not inserted between
  them). *Verify (positive, wrap-robust):* flatten the built file and assert the adjacency present:
  `tr '\n' ' ' < stages/stage-8.md | grep -c 'verified = no\`. A defective check found this way'`
  == **1**. *Self-test (mandatory, stage 8):* run the same flattened grep against a scratch copy in
  which the two sentences have been reordered (the additive spans spliced between them) and confirm
  it returns **0** — the oracle fires on the exact defect the reviewer flagged.

## Notes on labels

No criterion is advisory: every item is a checkable gate. There is no numeric regression metric
(prompt-doc change, no baseline) → stage 8 is **conformance-only** and says so.

## Dogfood self-test plan for the wrap-robust grep oracles (C3, C10)

Per the rule being added, every grep oracle over prose is **flattened first** and shown able to
**fire on a known-violating input** before its pass is trusted:
- **C3** — `tr '\n' ' ' < stage-8.md | grep -c 'fails against the unguarded version'` == 1.
  *Self-test:* same flattened grep on a scratch copy with the clause deleted → **0** (fires).
- **C10** — `tr '\n' ' ' < stage-8.md | grep -c 'verified = no\`. A defective check found this way'`
  == 1. *Self-test:* same grep on a scratch copy with the two sentences reordered → **0** (fires).
- **C8** — exact-pattern single-hit count (`grep -rn 'unreviewed check'` → 1 hit); no degenerate
  self-test needed because the pattern is contiguous and exact (not an absence sweep).
All self-tests run at stage 8 and are recorded in `8-harness.md`. Each replaces a fragile or
degenerate check that the round-1 red-team would (and did) reject.
