# 1.5 — Acceptance criteria

The change is a prose edit to two methodology docs. There is no measurable prior "behavior"
metric, so **stage 8 is conformance-only** (no stage-0 baseline). The conformance oracle has
two kinds of check: **text/structure inspection** of the edited docs, and one **execution
(replay)** check that is the load-bearing one — it proves the new guard would actually have
caught the bug it was written for, rather than merely reading well.

| # | Criterion | Gating? | How verified (stage 8) |
|---|---|---|---|
| **C1** | The guard is added in **all** load-bearing locations, so it fires end-to-end in the loop: (a) a Core Principle entry, (b) a **stage-2 plan-completeness deliverable** (enumerate accessors + state guard scope — the cheapest catch, per "instrument before you build"), (c) a red-team charter lens trigger used at stages 3/6, (d) a mandatory stage-1.5 criterion rule **and** a stage-8 execution requirement. | **gating** | Locate each part in the edited `METHODOLOGY.md`; confirm SKILL.md's operating procedure (stages 1.5 and 3/6) references the trigger. A guard in only one place does not fire — partial insertion is exactly the "deferred gating criterion" failure. |
| **C2** | The red-team lens explicitly requires **(i) enumerating every concurrent reader/writer** of the shared state and **(ii) challenging the guard's *scope*** (which accessors it does and does not cover) — not merely "verify the lock is correct." | **gating** | Text of the lens names both the enumeration step and "the guard's scope is a claim to challenge" (or equivalent). This is the exact step the compaction gates skipped. |
| **C3** | **REPLAY A/B (execution).** With the guard in its charter, a cold reviewer given the **same source the real stage-3 had** surfaces the `ingest_turn` lock-free lost-update race — and *without* the guard, on the same source, it does not (or is materially less likely to). The guard must be shown to **change the outcome**, not just to co-occur with a catch. | **gating** | Run the compaction case through a **cold subagent** (no access to this run) given the compaction `2-plan.md` **plus the concurrency-relevant source the real gate had** (`server.py` cadence-on-a-thread, `engine.py` lock-free `ingest_turn`, `supervisor.py` tick) — the same `redteam_context` where the guard actually fires. **Treatment:** charter includes the new guard text → must independently flag the lock-free concurrent writer / lost update. **Control (negative):** the documented historical miss stands as the paired control — the real stage-3 had that source and did **not** flag it; optionally re-run one stock-charter reviewer to reconfirm. Pass = treatment catches it **and** control (historical or fresh) does not. Fail = guard doesn't move the outcome → not representative. This is "a control must exhibit the phenomenon first," done as a real A/B rather than a reduced-context probe. |
| **C4** | The trigger is **negatively scoped** to avoid over-firing: it states it applies only when a change *introduces or modifies concurrent access to shared mutable state*, and **not** to ordinary single-threaded / name-bound code — mirroring the position lens's "not ordinary code" carve-out. | **gating** | Text presence of the negative scope in both the principle and the charter trigger. |
| **C5** | No coherence regression: the new material is consistent with the severity model, the two-layer split, and does **not** duplicate or contradict the position-sensitivity section. | advisory | Stage-6 cold review reports no logical/factual inconsistency introduced. *Advisory:* coherence is a real quality but minor-in-place fixable, not a ship blocker. |
| **C6** | Both copies — live `~/.claude/skills/guarded-change/{METHODOLOGY,SKILL}.md` and source `~/Desktop/claude-code-skills/Guarded_change/{METHODOLOGY,SKILL}.md` — are updated **identically**. | **gating** | `diff` the two copies of each changed file → empty. A source-only or live-only edit leaves the running skill wrong (dragonfly "edit skill AND source" lesson). |

**Position-sensitive-assembly note (self-referential irony, handled):** the methodology doc is
prose, not a behavior-by-position assembly in the technical sense, so the position lens does
not strictly apply. **But** the guard only *fires* if it lands in the three specific locations
(C1) — a placement requirement. C1 discharges it by execution (locate all parts), so a guard
that is present-but-misplaced fails, exactly as a displaced position-sensitive block would.
