# 1.5 — Criteria (the accept bar stage 8 checks)

All criteria below are **gating** unless labelled advisory. Verification means = how stage 8
proves it (by execution/inspection, not by re-reasoning about the text).

## Content criteria (the lens itself)

- **C1 (gating) — Fifth lens present & named.** After the change, `stages/charter.md`'s numbered
  lens list contains a fifth, distinctly-named **Fidelity** lens. *Verify:* the list enumerates
  1–5 and item 5 is the fidelity lens. Automated: `grep`.
- **C2 (gating) — Lens carries the four required elements.** The fidelity lens text (list entry +
  its discipline bullet) states all four of: (a) pin loaded operational terms to their concrete
  mechanism; (b) proxy substitution ⇒ UNTRUSTED until owner confirms; (c) an inherited
  definition (prior artifact / memory note) is a claim to re-verify, not a spec; (d) an
  earned-clean guard (a clean fidelity verdict must name the terms + pinned mechanism, else
  un-run). *Verify:* human-judged rubric (judge = the run operator) — each of (a)–(d) is
  locatable in the added text. Pass = all four present.
- **C3 (gating) — Unconditional, not conditional.** The lens is added as a core (always-run)
  lens, NOT gated behind a structural trigger the way the position/concurrency bullets are.
  *Verify:* inspection — the lens sits in the numbered core list; its discipline bullet has no
  "fires only when…" trigger clause. Rationale is recorded in the plan.
- **C4 (gating) — Lenses 1–4 and every `(lens 4)`/"lens 4" token are byte-unchanged.** The append
  must not alter the wording/numbering of the existing four lenses or **any** reference that names
  a lens by number — the `(lens 4)` tags on the position/concurrency discipline bullets in
  `charter.md` (L56, L67) **and** the "lens 4 un-run" / `(lens 4)` tokens in `stage-3.md` (L21,
  L29). *Verify:* `git diff` shows no edits inside lenses 1–4 or to any lens-number token in
  `charter.md` or `stage-3.md`. (Broadened from charter-only per red-team G1.)

## Cross-doc consistency criteria (the "four"→"five" surface)

- **C5 (gating) — Every count reference now positively reads "five"; no stale "four" survives —
  markdown/line-wrap-robust.** Redesigned per red-team BLOCKER F1: a bare `grep 'four…lens'` is a
  broken oracle because it misses `charter.md:16` (`four **separate** lenses` — `**` emphasis
  exceeds any small length bound) and `SKILL.md:66-67` (`four`\n`lenses` — a line-based grep can't
  span the wrap). C5 is therefore verified two ways, both required:
  - **(a) POSITIVE per-location assertion** — each of the seven post-edit strings is present
    exactly once at its location (whitespace/wrap-tolerant): charter `five **separate** lenses`
    (L16) and `five lenses + the unconditional discipline` (L46); METHODOLOGY `five lenses +
    discipline` (L82); SKILL the wrapped `cold subagent, five⎵lenses` (L66-67, matched
    newline-tolerantly with a **portable** flatten — `tr '\n' ' ' <SKILL.md | grep -o 'subagent,
    five *lenses'` — no `grep -P/-z`, so it holds on BSD/macOS grep too); stage-3 `five lenses +
    the discipline` (L6) and `five lenses + evidence discipline` (L15); stage-6 `five lenses + the
    discipline` (L6). All seven must be found.
  - **(b) NEGATIVE robust sweep** — flatten newlines + strip `**` markers, then grep for any
    residual "four…lens": `for f in SKILL.md METHODOLOGY.md stages/charter.md stages/stage-3.md
    stages/stage-6.md; do tr '\n' ' ' <"$f" | sed 's/\*\*//g' | grep -oiE 'four[a-z ]{0,15}lens'
    && echo "STALE:$f"; done` → **zero** STALE hits. **Self-test (mandatory stage-8 step):** run
    the *same* sweep against the pre-change images (`git show HEAD:<file>` piped through the same
    `tr … | sed … | grep`) and confirm it FIRES on all five files — a sweep that stays silent on
    the known-stale baseline proves nothing. Go-silent only after all seven are fixed.
- **C6 (gating) — The diff changes exactly the known count spots plus the charter additions, and
  nothing else.** The count word is updated at `charter.md` (×2), `METHODOLOGY.md` (×1), `SKILL.md`
  (×1), `stage-3.md` (×2), `stage-6.md` (×1); `charter.md` additionally gains the lens-5 list entry
  and the fidelity discipline bullet (E1/E2); the spot-verify bullet gains the M1 clause. *Verify:*
  `git diff` hunks = exactly {those seven count edits} ∪ {the three charter additions} and no
  other change. (Parenthetical restored per red-team L1 so an operator reading this row alone
  cannot over-tighten and false-fail.)

## Standing self-check criteria (mandatory when the artifact is this skill)

- **C7 (gating) — live copy == source copy.** After the build, `diff -r` of the installed skill
  (`~/.claude/skills/guarded-change/`) against the source (`~/Desktop/claude-code-skills/
  Guarded_change/`, excluding repo-only `README.md`, `*.companion.md`, `changes/`) is empty.
  *Verify:* `diff` exit 0.
- **C8 (gating) — cross-doc rule consistency.** Every rule stated in more than one place stays
  consistent after the edit. Here the only multiply-stated fact touched is the **lens count** —
  covered by C5/C6. No *other* rule text is altered. *Verify:* `git diff` review confirms only
  count words + the charter additions changed; no rule semantics elsewhere.
- **C9 (gating) — behavior-preservation for a position-sensitive assembly (position lens).** The
  skill docs are prompts (a position-sensitive assembly). The change is an **append** of a new
  lens at the end of the list + a new discipline bullet + in-place word swaps. Behavior to
  preserve: **the existing four lenses and every discipline/provenance/conditional-lens rule
  still read and apply exactly as before**; the only behavioral *addition* is that a red-team now
  also runs the fidelity lens. *Verify by execution:* (i) C4 (lenses 1–4 byte-unchanged) + (ii) a
  stage-6 cold reviewer, reading the edited charter, confirms it can still articulate lenses 1–4
  and the position/concurrency triggers unchanged AND now also the fidelity lens — i.e. the added
  block did not displace or reword any load-bearing neighbor. (No probabilistic rate here: the
  "behavior" of a prompt-doc lens is that a reader can enumerate and apply it; the reviewer
  enumeration is the execution.)

## Notes on labels

No criterion is advisory: every item above is a checkable gate. There is no numeric regression
metric (prompt-doc change, no baseline) → stage 8 is **conformance-only** and says so.
